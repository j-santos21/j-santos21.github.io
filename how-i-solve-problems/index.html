How I Solve Operational Problems

The pattern behind the three case studies in this portfolio · Jason Santos

Three different problems — a dashboard, an onboarding process, a release testing pipeline — but the same underlying approach shows up in each one. This page names that approach directly, so it doesn't have to be inferred from reading all three.

1

Notice the friction directly, not through a report

None of these problems showed up as an assigned project. They showed up as something that kept costing time or attention week after week — three untracked signals competing for focus, an onboarding process that some agencies visibly outgrew and others visibly drowned in, a defect that wouldn't have thrown an error if I hadn't been looking for it.

Every case study in this portfolio started this way, not from a request to "build a dashboard" or "improve onboarding."
2

Diagnose before building

The instinct is to start building a fix immediately. The better move is figuring out what's actually different about each case first — what does this specific agency need, what does this specific workflow actually depend on — because a fix built for an average case is wrong for almost everyone in a specific, predictable way.

See: the first-call diagnosis in the onboarding framework, and testing workflows in sequence rather than fields in isolation in the release testing case study.
3

Consolidate scattered signals into one place

Problems that live in three different places — memory, a spreadsheet, and nowhere — don't get solved by getting better at any one of them. They get solved by putting them somewhere they can be compared against each other, so attention goes where the data points, not where the loudest account happens to be.

See: the portfolio dashboard, which consolidated engagement, onboarding, and renewal risk into a single weekly view.
4

Build the smallest system the constraint allows

No engineering support, no dedicated tooling budget, and a day job that didn't pause while the fix got built — every one of these had to work inside those limits, which meant starting with the simplest version that solved the actual problem, not the most complete one.

See: starting in Excel before Power BI, and a five-minute diagnostic call instead of a formal intake system, in the dashboard and onboarding case studies.
5

Route decisions by business risk, not technical severity

Not every issue deserves the same response. The question that actually matters isn't "how big is this bug" or "how far behind is this agency" — it's what happens downstream if this doesn't get addressed. A rate defect that could reach a real quote gets escalated differently than a cosmetic issue. A renewal account with a declining trend gets a call before an account that's merely quiet.

See: the escalation path in release testing, and the renewal-risk flagging logic in the dashboard case study.
6

Ship it, use it, and let real behavior reveal what's wrong with it

None of these were designed perfectly the first time, and I don't think that's a flaw in the approach — a system that hasn't been used yet is still a guess. The tier assignment that should have been revisited partway through onboarding, the risk thresholds that were set by eye instead of validated against outcomes: those are things you only find by running the system against real cases, not by planning harder up front.

See the Tradeoffs & Lessons Learned section in each case study — that's this step, made explicit.
7

Document it so it works after you're gone

A fix that only works while you're the one running it isn't actually a system — it's a habit. The last step is making the logic explicit enough that someone else can run it without you, which is also the test of whether you actually solved the underlying problem or just got good at working around it yourself.

See: the onboarding framework, formally adopted org-wide as the standard playbook after departure.
Why this matters more than any single artifact: the dashboard, the onboarding framework, and the release testing process are three different outputs of the same process. The process is the actual skill; the artifacts are proof it was applied consistently, not three unrelated projects that happened to work out.
