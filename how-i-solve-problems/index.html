The Framework

How I Think

A seven-step loop for turning ambiguity into systems someone else can run · Jason Santos

A dashboard, an onboarding process, a release testing pipeline. Three different problems, one approach. This page names it directly so it doesn't have to be inferred from reading all three case studies. The loop isn't specific to operations - it's how I work through anything ambiguous; the case studies are just the documented evidence. It runs AI-assisted end to end these days, faster than it used to be, but the steps haven't changed.

1Notice→ 2Diagnose→ 3Consolidate→ 4Build small→ 5Route by risk→ 6Ship & learn→ 7Document

A fix built for the average case is wrong for almost everyone. The job is figuring out what's actually different.The thesis behind every step below.

1

Notice the friction directly, not through a report

None of these problems arrived as assigned projects. They showed up as something that kept costing time or attention week after week: three untracked signals competing for focus, an onboarding process some agencies outgrew and others drowned in, a defect that never threw an error.

Every case study on this site started this way. Nobody asked me to "build a dashboard" or "improve onboarding."
2

Diagnose before building

The instinct is to start fixing immediately. The discipline is holding off until you know what makes this case different from the last one that looked just like it.

See: the first-call diagnosis in the onboarding framework, and testing workflows in sequence rather than fields in isolation in release testing.
3

Consolidate scattered signals into one place

Problems that live in three different places - memory, a spreadsheet, and nowhere - don't get solved by getting better at any one of them. They get solved by putting them somewhere they can be compared, so attention goes where the data points, not where the loudest account happens to be.

See: the portfolio dashboard, which merged engagement, onboarding, and renewal risk into a single weekly view.
4

Build the smallest system the constraint allows

No engineering support, no tooling budget, and a day job that didn't pause while the fix got built. Every one of these had to start as the simplest version that solved the actual problem, not the most complete one.

See: Excel before Power BI, and a five-minute diagnostic call instead of a formal intake system.
5

Route decisions by business risk, not technical severity

The question that matters isn't "how big is this bug" or "how far behind is this agency" - it's what happens downstream if nothing changes. A rate defect that could reach a real quote escalates differently than a cosmetic issue. A renewal account with a declining trend gets a call before an account that's merely quiet.

See: the escalation path in release testing, and the renewal-risk flags in the dashboard.
6

Ship it, use it, and let real behavior reveal what's wrong with it

None of these were designed perfectly the first time, and that's not a flaw in the approach: a system that hasn't been used yet is still a guess. The tier assignment that should have been revisited mid-onboarding, the risk thresholds set by eye instead of validated against outcomes - you only find those by running the system against real cases.

The Tradeoffs & Lessons Learned section in each case study is this step, made explicit.
7

Document it so it works after you're gone

A fix that only works while you're running it isn't a system, it's a habit. The last step is making the logic explicit enough that someone else can run it without you - which is also the test of whether you solved the problem or just got good at working around it.

See: the onboarding framework, formally adopted org-wide as the standard playbook after my departure.

Where AI Fits in This Loop Now

The framework predates my AI workflow. What AI changed is the cycle time of each step, and that compounding speed is the difference between running this loop once a quarter and running it continuously:

  • Diagnose faster. I use Claude as a sparring partner on the diagnosis itself: describe the friction, argue about root causes, and get my own reasoning stress-tested before I commit to a build.
  • Prototype in hours, not weeks. The first testable version of a system - a tracker, a workflow, this site - now exists the same day the idea does. Step 4 used to be the bottleneck; it isn't anymore.
  • Ship-and-learn cycles compress. When iteration is cheap, step 6 stops being painful. Revising a system based on real behavior no longer competes with the day job.
  • Documentation stops being the step that gets skipped. Step 7 historically dies under time pressure. Drafting docs with AI in the loop means the logic gets captured while it's fresh, then edited by the person who owns it.

None of this replaces the judgment in steps 1, 2, and 5. Noticing the right friction, diagnosing what's actually different, and knowing what risk justifies escalation - that's still the skill. AI just removes the excuse for the loop running slowly.

What doesn't copy

You could adopt the same AI tools I use in an afternoon. The judgment underneath them is what took years.

The tools got dramatically faster over the last year. What didn't come from the tools: the instinct for which unowned problem is worth solving, the discipline to diagnose before building, and the habit of documenting a fix so it outlives you. Give two people the same experience and the same AI, and you still get different results, because the loop on this page is a way of seeing problems, not a tool you install. That's the part that doesn't replicate.

Why this page matters more than any single artifact: the dashboard, the onboarding framework, and the release testing process are three outputs of the same process. The process is the skill; the artifacts are proof it was applied consistently. If you remember one thing from this site, make it this: I turn ambiguity into systems that scale past me.
