Case Study — Technical Implementation & QA

Platform Release Testing & Cross-Functional Implementation

UAT and regression testing on a live, OCC/FDIC-supervised financial SaaS platform at Wintrust Financial Corporation · Jason Santos

Problem
Platform enhancements deployed into a live financial system with no dedicated QA function — a missed defect in a pricing or rate workflow could reach real agencies and real quotes.
Solution
Ran structured UAT and regression testing ahead of each release, working financial workflows end-to-end, and escalated defects through a defined path to engineering before anything reached production.
Result
Caught defects pre-production, including a rate-configuration sequencing bug that could have let quotes ship below the authorized buy-rate floor — flagged and held before it ever reached an agency.
Why this matters: this wasn't an internal tool — it was the platform agencies used to originate real quotes inside a regulated banking environment. A defect that reached production wasn't a bug ticket, it was a compliance exposure and a client-trust problem, and there was no dedicated QA team standing between engineering and the live system. Testing was part of the operational role, not a separate function.

Background

FirstInSite, Wintrust's enterprise financial SaaS platform, ran the workflows that priced and processed premium finance transactions across a $200M portfolio — policy entry, exposure calculations, pricing overrides, and Salesforce integrations. Enhancements were deployed on a recurring but uneven cadence, typically landing in Sunday maintenance windows. There was no dedicated QA team between engineering and production; account-side staff who understood how the workflows were actually used in practice were part of the testing loop before releases went live.

The Problem

Financial workflows on a regulated platform don't fail loudly. A defect in a pricing or rate field doesn't usually throw an error — it produces a wrong number that looks plausible, and by the time anyone notices, it may already be in a quote an agency has seen. Testing had to catch things that wouldn't announce themselves, using knowledge of how the workflow was supposed to behave, not just whether a page loaded.

Constraints

No automated regression suite — every test was run manually, field by field, workflow by workflow. No dedicated QA headcount, which meant testing depended on people who understood the business logic well enough to know where problems tended to hide, layered onto an already full account-management role. And the release cadence itself was a constraint: enhancements landed in a defined maintenance window, which compressed how much time existed between finding a problem and deciding whether to hold the release.

My Approach

Ahead of each release, I worked the affected financial workflows start to finish the way an agency actually would — policy entry, exposure calculations, pricing overrides — rather than testing fields in isolation. That mattered because the platform's underlying API had dependencies between certain fields: some had to be entered and processed in a specific order, and testing them one at a time in isolation wouldn't surface what only showed up when they interacted in sequence.

The Buy-Rate Sequencing Defect

The platform's rate configuration workflow had three related fields that had to be entered and processed independently, in a specific order — agency sell rate, referral fee (a dollar figure), and agency buy rate — because of how the underlying API validated them. Entered out of the expected sequence, the API would break silently: values would reset rather than throw a visible error.

I found it by testing the rate configuration workflow in the order an agency would actually use it, not by testing each field in isolation — the break only showed up when the three fields interacted in sequence. Left unflagged, the failure mode meant a quote could ship with a buy rate below the authorized floor: a downstream conversation about whether to honor a rate that should never have been offered, or damage the agency relationship walking it back after the quote was already out. I flagged it before release, and the release held until engineering fixed the sequencing issue.

Why It Worked — the Reasoning

The mechanism isn't "I tested the release" — it's that testing the workflow in sequence, not the fields in isolation, is what exposes interaction bugs. Each of the three rate fields validated correctly on its own; a test script checking each field independently would have passed all three and shipped the defect anyway. The bug only existed in the space between the fields — in the order they were processed — which meant it could only be caught by someone testing the way a real user actually moves through the workflow, not by someone checking whether each individual input accepted valid data.

That's the broader pattern behind the escalation structure too: routing defects to BAs first, who understood both the technical system and the client-facing consequence, meant issues got framed in terms of business risk (a below-floor rate reaching an agency) rather than just a technical ticket — which is what justified holding a release inside a compressed Sunday deployment window instead of letting it ship and fixing it after the fact.

Tradeoffs & Lessons Learned

What would I redesign today?

There was no formal, documented regression checklist covering the workflows that had proven fragile before — testing depended on personal, tacit knowledge of where problems tended to hide, mine and other testers'. I'd build a living checklist that captured every known-fragile interaction (like the rate-field sequencing) so catching it didn't depend on one person remembering it existed.

What data would I collect now that I couldn't then?

Defects found in testing weren't logged in a way that showed patterns over time — I couldn't point to which workflows produced the most catches or whether certain types of releases were riskier than others. I'd track every defect by workflow area and root cause, so a recurring problem like field-sequencing dependencies would surface as a systemic risk rather than get rediscovered by chance each time.

What resistance did this face?

The honest tension was time, not disagreement — flagging a hold on a compressed Sunday deployment window meant the finding had to be clearly justified and communicated fast, not just "this looks off." A vague concern wouldn't stop a release; a specific, reproducible sequence would.

Where's the limit of this design?

It worked because of accumulated familiarity with exactly where this platform tended to break, built over years of running the same workflows. That kind of tacit expertise doesn't transfer easily to a new hire and doesn't scale past a small, tenured team — which is exactly the gap a documented checklist and automated regression coverage would close.

If I Rebuilt This in 2026

The core gap named above — that catching this class of bug depended on one person's accumulated knowledge of where the platform broke — is exactly what AI-assisted testing tooling is positioned to close today:

  • AI-assisted regression suite generation. Feed known-fragile workflows (like the rate-field sequencing dependency) into a test-generation tool that builds and maintains automated coverage for field-interaction bugs, not just single-field validation — the exact category a script checking fields in isolation would have missed.
  • Anomaly detection on QA runs. Instead of a tester needing to visually notice a value silently resetting, an automated diff on before/after field states during test runs would flag the exact symptom that surfaced the buy-rate defect, automatically.
  • A living defect knowledge base. An LLM-summarized log of every defect found, tagged by workflow and root cause, so patterns get surfaced proactively — directly answering the "what data would I collect now" gap above.
  • Severity-routed escalation drafts. AI-drafted summaries that route to the right stakeholder — BA manager, sales EVP, or the SVP - BLIS Officer — based on defect severity and business risk, cutting the time between a catch and a decision inside a fixed deployment window.
